# Weekly / Monthly Review

mycraft의 핵심 제품 기능 — 사용자 기록을 읽고 다음 목표를 추천한다.

## 입력 확인

리뷰 시작 전 데이터 가용성을 확인한다.

| 데이터 양 | 가능한 분석 |
|----------|----------|
| 1주 미만 | "데이터 누적 단계" 안내. 추세 추론 안 함. 기록 습관 팁만. |
| 1~4주 | 단순 빈도/시간 합계 + 첫 패턴 가설 (확신 낮음 표시). |
| 4주~3개월 | 주간 비교 + 추세 + 다음 주 목표 추천. |
| 3개월+ | 월간 비교 + 계절성 + 장기 추세 + 다음 달 목표 추천. |

## 분석 프레임워크

### 1단계: 수치 요약 (객관)

사용자가 정의한 활동 카테고리별로:
- 이번 기간 횟수 / 총 시간 / 평균 강도.
- 같은 길이의 직전 기간 대비 증감 (%).
- 가장 활발한 요일 / 시간대.

표 예시:
```markdown
| 활동 | 이번 주 | 지난 주 | 증감 | 활발 요일 |
|------|--------|--------|------|---------|
| 운동 | 3회/3.5h | 4회/4h | -25%/-12% | 화/목 |
| 독서 | 5일/4h | 4일/3h | +25%/+33% | 평일 저녁 |
```

### 2단계: 패턴 도출 (해석)

수치에서 의미 있는 패턴을 찾는다.

- **유지 패턴** — 4주 이상 안정적인 활동 (예: "화/목 운동 패턴이 6주째 유지됨").
- **상승 패턴** — 점진적 증가 (예: "독서 시간이 4주간 매주 +15%").
- **하락 패턴** — 점진적 감소 (예: "주말 활동량이 3주째 감소").
- **이상치** — 갑작스런 급증/급감 (사용자에게 원인 묻기).

**중요:** 1~2주 데이터로 패턴이라 부르지 않는다. 최소 4주 일관성이 있어야 패턴.

### 3단계: 다음 기간 목표 추천

원칙:
- **현재 패턴 + 작은 도전.** 안정 패턴은 유지, 가능한 부분만 +1 단위로 확장.
- **2단 옵션 제시.** 1차(이상적) + 2차(부담 시 축소판). 사용자가 거절할 수 있게.
- **새 활동 도입은 신중히.** 기존 패턴이 무너지지 않을 때만.

추천 형식:
```markdown
## 다음 주 추천

**1차 (이상):**
- 운동 4회 (화/목/토 + 일요일 가벼운 산책 추가)
- 독서 평균 1h/일 유지

**2차 (부담 시):**
- 운동 3회 (현재 화/목 패턴 유지)
- 독서는 현 수준 유지

선택 가능한 옵션입니다. 더 줄이고 싶으면 알려주세요.
```

## 산출물

`_workspace/review_{period}.md` 형식 (period = `wk-2026-W18` 또는 `mo-2026-05`):

```markdown
# Review: {기간 라벨}

## 1. 수치 요약
{표}

## 2. 패턴
{유지/상승/하락 패턴 + 이상치}

## 3. 다음 기간 추천
**1차 (이상):**
{...}

**2차 (부담 시):**
{...}

## 4. 회고 노트 (선택)
{사용자가 추가한 메모 + Sage의 관찰}

## 5. 변경 이력 기록 후보
{이번 리뷰에서 AGENTS.md 변경 이력에 추가할 사항}
```

## 사용자 데이터 위치 가정

기술 스택이 정해진 후 갱신:

- **DB 직접 조회** — Architect의 결정에 따라 SQLite/PostgreSQL.
- **로컬 파일** — JSON/CSV 형식으로 export된 기록.
- **사용자 직접 입력** — DB가 없으면 사용자가 채팅으로 직접 알려주는 형식 폴백.

스택 미정 상태에서는 사용자에게 직접 데이터를 묻는 폴백을 사용:
"지난 주 활동을 알려주세요 (활동 / 횟수 / 시간)."

## 후속 작업

리뷰 결과를 사용자가 수락하면:
1. 다음 기간 목표를 (스택이 갖춰지면) DB에 저장.
2. AGENTS.md 변경 이력에 한 줄 추가 (Sage가).
3. 목표 변경이 신규 기능을 요구하면 Quest Master에게 SendMessage로 인계.

## 안티패턴

- "운동을 더 하세요" 같은 일반론. → 항상 구체 수치 + 일정.
- 데이터 부족한데 패턴 단정. → "확신 낮음" 명시.
- 사용자가 거절할 수 없는 강제적 어조. → 항상 옵션 제시.
- 부정적 평가 우선. → 잘 된 것 먼저 짚고, 개선점은 다음.
