# Quest: 시즌 챌린지 운영 체계

> 작성: Quest Master | 2026-05-06 | 입력: 사용자 피드백 | 다음 페르소나: Artisan

## 0. 현재 문제

현재 mycraft는 "오늘 한 일을 사용자가 직접 고르고 기록한다"가 중심이다. 이 구조는 빠르게 만들 수 있지만 제품 경험은 일반 기록 앱과 크게 다르지 않다.

- 사용자가 눌러야 할 것이 많다.
- 앱이 오늘 무엇을 해야 하는지 제안하지 않는다.
- XP, 레벨, 스탯은 기록 뒤에 붙는 보상 스킨에 가깝다.
- 월간, 주간, 일간 계획이 하나의 게임 루프로 연결되지 않는다.
- "직관적인 카드 목록"이라서 안전하지만, 사용자가 싫어하는 평범한 생산성 앱의 인상을 준다.

핵심 전환:

> mycraft는 기록 앱이 아니라, 사용자의 한 달을 하나의 시즌으로 운영하는 개인 챌린지 게임이어야 한다.

Today 화면은 최상위 화면이 아니다. 시즌이 내려준 오늘의 미션을 수행하고 결과를 보고하는 하위 화면이다.

## 1. 제품 원칙

### P1. 사용자가 계획을 매번 만들지 않는다

사용자는 "이번 시즌 방향"만 선택한다. 시스템은 월간 시즌, 주간 에피소드, 일간 미션을 자동으로 생성한다.

### P2. 기록보다 챌린지가 먼저다

기존 흐름:

```
기록한다 -> XP를 받는다 -> 통계를 본다
```

새 흐름:

```
시즌을 시작한다 -> 오늘의 미션이 열린다 -> 수행한다 -> 시즌 판도가 바뀐다
```

### P3. 직관성은 낮추되 이해 가능성은 유지한다

흔한 to-do UX를 피한다. 대신 "시즌", "작전", "제약", "보상 선택", "판도 변화" 같은 구조를 쓴다. 다만 사용자가 길을 잃으면 안 된다. 매 화면에는 다음 행동 1개가 분명해야 한다.

### P4. 클릭 수를 줄인다

기록 단위는 카테고리 카드 여러 개가 아니라 "오늘의 미션 카드"다.

- 기본: 1탭 완료
- 수치 입력: 카드 안에서 즉시 조정
- 세부 편집: 길게 누르기 또는 더보기
- 하루 종료: 자동 요약, 수동 마감은 선택

## 2. 핵심 메커닉

### 2.1 Season

한 달 단위의 큰 챌린지. 시즌은 사용자가 이번 달에 바꾸고 싶은 방향을 담는다.

예:

- "체력 회복 시즌"
- "딥워크 복구 시즌"
- "생활 리듬 재정렬 시즌"
- "콘텐츠 생산 시즌"

필드 초안:

| 필드 | 설명 |
|------|------|
| title | 시즌 이름 |
| intent | 이번 달에 바꾸고 싶은 상태 |
| duration | 시작일, 종료일 |
| primary_stat | 시즌 주 스탯 |
| secondary_stat | 보조 스탯 |
| difficulty | easy, normal, hard, brutal |
| rules | 시즌 제약 1~3개 |
| win_condition | 시즌 승리 조건 |
| failure_condition | 시즌 실패 조건 또는 경고 조건 |

### 2.2 Challenge

시즌 안의 구체적 목표. 챌린지는 단순 목표가 아니라 규칙과 긴장을 가진다.

예:

- "주 4회 운동"이 아니라 "Vitality를 매주 4칸 이상 점령"
- "매일 독서"가 아니라 "7일 중 5일, 밤 11시 이전 Intelligence 미션 완료"
- "딥워크 10시간"이 아니라 "주간 집중 구역 3개 확보"

챌린지 구성:

| 구성 | 설명 |
|------|------|
| target | 달성하려는 행동 또는 결과 |
| cadence | 월/주/일 반복 구조 |
| pressure | 실패를 만들지는 않지만 긴장감을 주는 조건 |
| reward | XP 외에 해금되는 선택지 |
| adaptation | 지난 기록에 따라 다음 계획을 조정하는 방식 |

### 2.3 Campaign Plan

월간 계획은 시즌 전체의 지도다.

- 1개월을 4~5개 주간 에피소드로 나눈다.
- 각 주는 다른 역할을 가진다.
- 초반은 진입, 중반은 밀도, 후반은 정산과 보정으로 간다.

예:

| 주차 | 역할 | 목표 |
|------|------|------|
| Week 1 | 정찰 | 현재 루틴의 실제 체력 파악 |
| Week 2 | 압축 | 안정적으로 반복 가능한 최소 루프 확정 |
| Week 3 | 돌파 | 가장 약한 스탯에 집중 부하 |
| Week 4 | 정복 | 시즌 승리 조건 달성 및 다음 시즌 재료 확보 |

### 2.4 Weekly Episode

주간 단위의 작전. 사용자는 매주 월요일에 "이번 주 작전"을 받는다.

주간 작전은 다음을 포함한다.

- 이번 주 핵심 챌린지 1개
- 보조 챌린지 1개
- 버려도 되는 행동 1개
- 위험 신호 1개
- 금요일 또는 일요일 리뷰 질문 3개

### 2.5 Daily Mission

일일 단위. 기존 Today의 대체 중심.

일일 미션은 사용자가 직접 카테고리를 골라 기록하는 것이 아니라, 시즌과 주간 작전에서 자동 생성된다.

예:

```
오늘의 미션
1. Vitality 확보: 운동 30분
2. Focus 점령: 딥워크 60분
3. Recovery 조건: 23:30 이전 마감
```

기록 방식:

- 미션 완료: 1탭
- 수치 조정: 미션 카드 안에서 +5분, +15분, 직접 입력
- 대체 수행: "다른 방식으로 완료" 선택
- 실패 처리: 실패 버튼이 아니라 "오늘 포기하고 내일 계획 보정"

## 3. 사용자 시나리오

### S1. 시즌 시작

1. 사용자가 앱을 연다.
2. 시스템이 "새 시즌을 시작할 시간"을 보여준다.
3. 사용자는 이번 달의 방향을 한 문장으로 입력한다.
4. 시스템은 시즌 후보 3개를 만든다.
5. 사용자는 하나를 고르고 난이도만 조정한다.
6. 시스템이 월간 캠페인, 이번 주 에피소드, 오늘의 미션을 생성한다.

성공 조건: 사용자가 3분 안에 시즌 시작을 끝낸다.

### S2. 하루 실행

1. 사용자가 `/today`에 들어온다.
2. 카테고리 목록이 아니라 "오늘의 미션 3개"가 먼저 보인다.
3. 사용자는 첫 미션을 완료한다.
4. XP가 뜨는 동시에 시즌 지도에서 해당 구역이 변한다.
5. 앱은 다음에 누를 행동 하나만 보여준다.

성공 조건: 일반적인 하루 기록이 3탭 이하로 끝난다.

### S3. 주간 보정

1. 일요일 밤 또는 월요일 아침, 시스템이 지난 주 결과를 요약한다.
2. 성공/실패가 아니라 "이번 주 판도"를 보여준다.
3. 시스템은 다음 주 미션 밀도를 자동 조정한다.
4. 사용자는 조정안을 그대로 수락하거나 난이도만 올리고 내린다.

성공 조건: 주간 계획 생성에 5분 이상 쓰지 않는다.

## 4. 보상 구조

XP는 계속 유지하지만 중심 보상이 아니다. 시즌 보상은 "다음 선택지가 바뀌는 것"이어야 한다.

### 즉시 보상

- 미션 완료 XP
- 주 스탯 상승
- 시즌 지도 구역 점령

### 주간 보상

- Episode clear
- 다음 주 보정권 1개
- 실패한 미션을 더 현실적인 미션으로 재생성

### 시즌 보상

- 시즌 칭호
- 다음 시즌 시작 보너스
- 사용자의 실제 패턴에 기반한 추천 목표
- "이번 달에 효과가 있었던 규칙" 저장

## 5. 추천/생성 규칙

데이터가 부족한 초기 버전은 AI 생성보다 규칙 기반이 낫다. 4주 이상 기록이 쌓이면 Sage가 추천 강도를 높인다.

### 0~1주 데이터

- 사용자가 입력한 시즌 의도 기반으로 템플릿 생성
- 난이도는 낮게 시작
- 미션은 하루 2~3개로 제한

### 2~3주 데이터

- 완료율이 80% 이상인 미션은 10~20% 강화
- 완료율이 40% 이하인 미션은 강도 하향 또는 시간대 변경
- 특정 요일 실패가 반복되면 그 요일 미션 수를 줄임

### 4주 이상 데이터

- 월간 시즌 추천 가능
- 주간 목표를 실제 평균치 기준으로 추천
- 약한 스탯을 보완하거나 강한 스탯을 밀어붙이는 두 가지 시즌 후보 제시

## 6. 성공 지표

| 지표 | 목표 |
|------|------|
| 하루 기록 완료 탭 수 | 평균 3탭 이하 |
| 사용자가 직접 새 카테고리를 고르는 빈도 | 일반 기록의 20% 이하 |
| 시즌 시작 완료 시간 | 3분 이하 |
| 주간 계획 수락/수정 시간 | 5분 이하 |
| 7일차 재방문 | 현재 Today 단독 대비 증가 |
| 사용자가 "오늘 뭐 하지?"라고 느끼는 빈도 | 앱 진입 시 첫 화면에서 해소 |

## 7. v1 구현 범위 제안

한 번에 시즌 전체 자동화를 구현하지 않는다. 다음 순서가 현실적이다.

### Milestone 1. Season Shell

- 현재 `/today` 위에 현재 시즌 영역 추가
- 시즌 이름, 남은 일수, 이번 주 작전, 오늘의 미션 3개 표시
- 기존 카테고리 기록은 미션 완료의 하위 동작으로 연결

### Milestone 2. Plan Generator

- 시즌 의도 입력
- 난이도 선택
- 월간/주간/일간 계획 템플릿 생성
- in-memory store 기준 구현 후 DB 승격

### Milestone 3. Weekly Review

- 완료율, XP, 스탯 변화를 기반으로 다음 주 미션 자동 조정
- "강화 / 유지 / 완화" 선택지만 제공

## 8. 다음 페르소나에게 넘길 결정

Artisan은 다음을 디자인해야 한다.

- 첫 화면을 `/today`가 아니라 `Season Command`로 재구성한다.
- Today는 "오늘의 미션 처리"에 집중한다.
- 카테고리 카드는 보조 입력으로 낮춘다.
- 월/주/일 구조가 한눈에 보이는 지도형 UI를 만든다.
- 직관적인 CRUD UI가 아니라, 시즌을 운영하는 콘솔처럼 보여야 한다.
